## 一、项目结构理解

### 1.1 解释 mini-agent 项目的目录结构

```
mini-agent/
├── pyproject.toml
├── src/
│   └── mini_agent/
│       ├── __init__.py
│       ├── main.py
│       ├── agent.py
│       ├── models.py
│       ├── tools.py
│       └── config.py
└── tests/
    ├── conftest.py
    ├── test_tools.py
    └── test_agent.py
```

**问题**：
1. 为什么使用 `src/` 布局？直接在根目录创建 `mini_agent/` 包有什么问题？
2. `__init__.py` 文件的作用是什么？可以省略吗？
3. `conftest.py` 文件的作用是什么？为什么放在 `tests/` 目录下？

**答案**：
1. `src/` 布局防止 `import mini_agent` 意外地导入当前目录的代码（而非安装后的包）。直接在根目录创建包，可能会导入未安装的本地代码。
2. `__init__.py` 标记目录为 Python 包。Python 3.3+ 有 namespace packages 可以省略，但不推荐。文件可以为空，但必须存在。
3. `conftest.py` 放置共享 fixture 和 hooks，自动被同级及子目录的测试发现。相当于自动 import 的公共模块。

## 二、Agent 核心理解

### 2.1 ReAct 循环流程

```python
for step in range(self.max_steps):
    messages = self._build_messages(history)
    response = await self.client.chat.completions.create(...)
    message = response.choices[0].message

    if not message.tool_calls:
        reply = message.content or ""
        return {"reply": reply, ...}

    # 执行工具调用
    for tc in message.tool_calls:
        result = self._execute_tool({...})
        history.append({"role": "tool", ...})
```

**问题**：
1. ReAct 循环的三个步骤是什么？
2. 为什么需要 `max_steps` 限制？
3. 工具调用的结果如何反馈给 LLM？

**答案**：
1. ReAct 循环：Reasoning（LLM 思考）→ Action（调用工具）→ Observation（获取结果）
2. 防止 Agent 陷入无限循环，控制成本和延迟。
3. 工具结果作为 `tool` 角色的消息加入历史，下次 LLM 调用时会看到这些结果。

---

### 2.2 工具注册机制

```python
TOOLS = [
    {
        "type": "function",
        "function": {
            "name": "calculate",
            "description": calculate.__doc__,
            "parameters": CalculateInput.model_json_schema(),
        },
    },
]

TOOL_FUNCTIONS = {
    "calculate": lambda args: calculate(args["expression"]),
}
```

**问题**：
1. `TOOLS` 和 `TOOL_FUNCTIONS` 的作用分别是什么？
2. 为什么使用 Pydantic 模型定义工具参数？
3. docstring 的重要性是什么？

**答案**：
1. `TOOLS` 是 OpenAI 格式的工具 schema，告诉 LLM 有哪些工具可用。`TOOL_FUNCTIONS` 是工具名到实现函数的映射，用于实际执行。
2. Pydantic 自动生成 JSON Schema，带类型和描述，确保参数校验。
3. docstring 就是 LLM 看到的工具描述，写清楚"什么时候用"和"参数含义"，LLM 靠它决定调用哪个工具。

### 2.3 流式输出实现

```python
async def stream(self, query, history):
    for step in range(self.max_steps):
        response = await self.client.chat.completions.create(...)
        message = response.choices[0].message

        if not message.tool_calls:
            reply = message.content or ""
            for char in reply:
                yield StreamEvent(type="token", content=char)
            yield StreamEvent(type="done")
            return

        # 工具调用
        for tc in message.tool_calls:
            yield StreamEvent(type="tool_call", name=tool_name, content=tool_args)
            result = self._execute_tool({...})
            yield StreamEvent(type="tool_result", name=tool_name, content=result)
```

**问题**：
1. 为什么使用 `AsyncGenerator` 而不是返回完整结果？
2. `StreamEvent` 的类型有哪些？
3. 如何在前端消费这些事件？

**答案**：
1. 流式输出可以立即返回部分结果，提升用户体验。用户可以看到 LLM 的思考过程和工具调用过程。
2. `StreamEvent` 类型：`token`（文本 token）、`tool_call`（工具调用）、`tool_result`（工具结果）、`done`（完成）、`error`（错误）
3. 前端使用 `EventSource` 或 `fetch` + `ReadableStream` 消费 SSE 事件。

## 三、工具实现

### 3.1 安全的 eval 实现

```python
def calculate(expression: str) -> str:
    allowed_names = {
        "math": math,
        "abs": abs,
        "round": round,
        "int": int,
        "float": float,
    }
    result = eval(expression, {"__builtins__": {}}, allowed_names)
    return str(result)
```

**问题**：
1. 为什么需要 `allowed_names` 白名单？
2. `{"__builtins__": {}}` 的作用是什么？
3. 这种实现还有哪些安全隐患？

**答案**：
1. 限制可用的函数和模块，防止恶意代码执行。
2. 清空内置函数，防止访问 `__import__`、`exec` 等危险函数。
3. 仍然存在安全隐患：可以访问对象的 `__class__`、`__subclasses__()` 等属性，可能逃逸沙箱。生产环境应使用更安全的表达式解析器（如 `ast.literal_eval` 或 `simpleeval`）。

```python
@pytest.fixture(autouse=True)
def mock_settings():

        mock.openai_api_key = "test-key"
        mock.openai_model = "gpt-4o"
        mock.openai_base_url = "http://localhost:8001"
        mock.max_agent_steps = 5
        mock.log_level = "DEBUG"
        yield mock
```

**问题**：
1. 为什么需要 `autouse=True`？
2. `yield` 的作用是什么？
3. 为什么 patch `"mini_agent.config.settings"` 而不是 `"config.settings"`？

**答案**：
1. `autouse=True` 让 fixture 自动应用到所有测试，无需显式声明。
2. `yield` 前是 setup 代码，`yield` 后是 teardown 代码。测试结束后会自动清理。
3. 必须 patch 函数被导入的地方，而不是定义的地方。测试代码导入的是 `mini_agent.config.settings`，所以要 patch 这个路径。

## 六、面试题

### 6.1 ReAct 循环的工作原理是什么？

**答**：ReAct 循环包含三个步骤：
1. **Reasoning**：LLM 分析用户问题，决定是否需要调用工具
2. **Action**：如果需要工具，LLM 返回 tool_calls，Agent 执行工具
3. **Observation**：工具结果作为 tool message 加入历史，LLM 再次分析并决定下一步

循环持续直到 LLM 给出最终回复或达到最大步数限制。

---

### 6.2 如何防止 Agent 陷入无限循环？

**答**：
1. 设置 `max_steps` 限制（如 10 步）
2. 检测重复工具调用（相同工具 + 相同参数）
3. 超时机制（整体超时 + 单步超时）
4. 设计明确的终止条件（LLM 返回最终答案而非工具调用）

---

### 6.3 流式输出的实现原理是什么？

**答**：
1. 使用 `AsyncGenerator` 逐步 yield 事件
2. FastAPI 的 `StreamingResponse` 将事件流转换为 SSE（Server-Sent Events）
3. 前端使用 `EventSource` 或 `fetch` + `ReadableStream` 消费事件
4. 事件类型包括：`token`（文本）、`tool_call`（工具调用）、`tool_result`（工具结果）、`done`（完成）

---

### 6.4 如何测试 Agent 的工具调用？

**答**：
1. Mock OpenAI 客户端，返回预定义的 `tool_calls`
2. 验证 Agent 是否正确执行工具并返回结果
3. 测试错误处理：Mock 工具抛出异常，验证 Agent 是否正确处理
4. 测试流式输出：收集事件流，验证事件类型和内容

---

### 6.5 生产环境部署 Agent 需要注意什么？

**答**：
1. **超时**：LLM 调用必须设超时（建议 30-60s）
2. **重试**：指数退避 + 最大重试次数（应对 429 限流）
3. **限流**：控制并发请求数（Semaphore / Redis 限流）
4. **降级**：LLM 不可用时的 fallback（缓存答案、转人工）
5. **监控**：延迟、错误率、token 消耗、工具调用成功率
