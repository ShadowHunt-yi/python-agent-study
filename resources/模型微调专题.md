# 模型微调专题

> 面试中高频出现的 LLM 微调技术，从原理到踩坑经验全覆盖

---

## 1. LLM 训练三阶段

```
预训练 (Pretraining)
  │  海量无标注文本，学习语言能力
  ▼
监督微调 (SFT)
  │  高质量指令-回复对，学习对话格式
  ▼
对齐 (Alignment)
  │  RLHF / DPO / RLAIF，学习人类偏好
  ▼
可用的 LLM
```

| 阶段 | 数据 | 目标 | 成本 |
|------|------|------|------|
| 预训练 | TB 级无标注文本 | 学习语言分布 | 数百万美元 |
| SFT | 万-百万级指令对 | 学习指令遵循 | 数千美元 |
| 对齐 | 人类偏好标注 | 学习安全/有用/诚实 | 数千-万美元 |

## 2. 参数高效微调（PEFT）

### 2.1 为什么需要 PEFT

全量微调（Full Fine-Tuning）的问题：

- **显存爆炸**：7B 模型全量微调需要 ~112GB 显存（参数 + 梯度 + 优化器状态）
- **灾难性遗忘**：微调后丢失预训练知识
- **存储开销**：每个任务一份完整模型副本

PEFT 只训练少量参数（0.1%-1%），效果接近全量微调。

### 2.2 LoRA（Low-Rank Adaptation）

**核心思想**：冻结原始权重 W，用低秩矩阵分解 ΔW = B × A 来近似更新

```
原始: h = W·x
LoRA: h = W·x + (B·A)·x

W ∈ R^{d×d}  (冻结，不训练)
A ∈ R^{r×d}  (可训练，r << d)
B ∈ R^{d×r}  (可训练，r << d)
```

**参数量对比**（以 d=4096 为例）：

| 方案 | 可训练参数 | 占比 |
|------|-----------|------|
| 全量微调 | 16.7M (4096²) | 100% |
| LoRA r=8 | 65K (4096×8×2) | 0.39% |
| LoRA r=16 | 131K (4096×16×2) | 0.78% |

**代码示例**（使用 PEFT 库）：

```python
from peft import LoraConfig, get_peft_model

config = LoraConfig(
    r=16,                          # 秩
    lora_alpha=32,                 # 缩放系数
    target_modules=["q_proj", "v_proj", "k_proj", "o_proj",
                     "gate_proj", "up_proj", "down_proj"],  # 所有线性层
    lora_dropout=0.05,
    bias="none",
    task_type="CAUSAL_LM",
)

model = get_peft_model(base_model, config)
model.print_trainable_parameters()
# trainable params: 41,943,040 || all params: 6,779,883,520 || trainable%: 0.62%
```

### 2.3 QLoRA

在 LoRA 基础上，将基础模型量化为 4-bit（NF4 格式），进一步降低显存：

```
LoRA:   7B 模型 → ~28GB 显存
QLoRA:  7B 模型 → ~18GB 显存（节省 ~33%）
```

**QLoRA 的代价**：训练速度慢 ~39%（反量化计算开销）

```python
from transformers import BitsAndBytesConfig

bnb_config = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_quant_type="nf4",        # NormalFloat4
    bnb_4bit_compute_dtype=torch.bfloat16,
    bnb_4bit_use_double_quant=True,    # 双重量化
)

model = AutoModelForCausalLM.from_pretrained(
    "meta-llama/Llama-3-8B",
    quantization_config=bnb_config,
)
```

### 2.4 其他 PEFT 方法对比

| 方法 | 原理 | 可训练参数 | 适用场景 |
|------|------|-----------|---------|
| **LoRA** | 低秩矩阵分解 | 0.1%-1% | 通用，最推荐 |
| **Prefix Tuning** | 在每层注入可训练前缀向量 | ~0.1% | 生成任务 |
| **Prompt Tuning** | 在输入前加可训练软提示 | ~0.01% | 超大模型（10B+） |
| **P-Tuning v2** | 每层加可训练提示 | ~0.1%-1% | NLU 任务 |
| **IA3** | 缩放激活值 | ~0.01% | 极端低资源 |

**面试回答要点**：LoRA 是当前工业界最主流的 PEFT 方法，因为它效果好、实现简单、推理时无额外开销（可合并回原权重）。

## 3. LoRA 实战踩坑经验

### 3.1 rank 和 alpha 的选择

```
常见错误：r=4, lora_alpha=8    → 表达能力不足
推荐配置：r=16, lora_alpha=32  → alpha = 2 × rank
         r=32, lora_alpha=64  → 复杂任务
```

**经验法则**：
- `lora_alpha / rank` 的比值决定了缩放系数，一般保持 alpha = 2 × rank
- rank 越大表达能力越强，但容易过拟合
- 简单任务（情感分类）：r=8 足够
- 复杂任务（代码生成、推理）：r=32 或更高

### 3.2 target_modules 的选择

```python
# 只训练注意力层（早期推荐，现在不够）
target_modules=["q_proj", "v_proj"]

# 训练所有线性层（当前最佳实践）
target_modules=["q_proj", "v_proj", "k_proj", "o_proj",
                 "gate_proj", "up_proj", "down_proj"]
```

**踩坑**：只训练 Q/V 层效果远不如训练所有线性层。Sebastian Raschka 的实验表明，all-layer LoRA 在相同 rank 下效果显著更好。

### 3.3 多 epoch 过拟合问题

```
常见错误：训练 10 epochs → 严重过拟合
推荐做法：1-2 epochs 足够
```

**原因**：LoRA 参数量少，数据量不大时很容易过拟合。如果 loss 在第 2 个 epoch 开始上升，立即停止。

**解决方案**：
- 用 early stopping
- 增加 `lora_dropout`（0.05-0.1）
- 减小 rank
- 增加训练数据量

### 3.4 优化器选择

```python
# AdamW（默认推荐）
optimizer = torch.optim.AdamW(model.parameters(), lr=2e-4)

# SGD（显存更省，但实际差异不大）
optimizer = torch.optim.SGD(model.parameters(), lr=1e-3, momentum=0.9)
```

**踩坑**：很多人以为 SGD 比 Adam 省很多显存。实际上对于 LoRA（参数量很少），优化器状态的差异可以忽略。直接用 AdamW 即可。

### 3.5 学习率范围

| 场景 | 推荐学习率 |
|------|-----------|
| LoRA 全量层 | 1e-4 ~ 3e-4 |
| LoRA 仅注意力 | 2e-4 ~ 5e-4 |
| QLoRA | 1e-4 ~ 2e-4 |
| 全量微调 | 1e-5 ~ 5e-5 |

**踩坑**：LoRA 的学习率通常比全量微调高一个数量级，因为可训练参数少、梯度信号弱。

## 4. RLHF（人类反馈强化学习）

### 4.1 三步流程

```
Step 1: SFT（监督微调）
  │  用高质量指令数据微调基础模型
  ▼
Step 2: 训练奖励模型（Reward Model）
  │  用人类偏好标注（A 比 B 好）训练打分模型
  ▼
Step 3: PPO 优化
  │  用奖励模型的分数作为 reward，用 PPO 算法优化策略
  ▼
对齐后的模型
```

### 4.2 奖励模型训练

```python
# 偏好数据格式
{
    "prompt": "解释量子计算",
    "chosen": "量子计算利用量子力学原理...",   # 人类偏好的回答
    "rejected": "量子计算就是很快的计算机...", # 人类不偏好的回答
}

# 训练目标：让 chosen 的分数高于 rejected
loss = -log(sigmoid(r_chosen - r_rejected))
```

### 4.3 Llama 2 的 RLHF 实践

Llama 2 的关键创新：

1. **两个奖励模型**：一个评 helpfulness，一个评 safety
2. **拒绝采样（Rejection Sampling）**：生成多个回复，只用最高分的进行训练
3. **Margin Loss**：`loss = -log(sigmoid(r_chosen - r_rejected - margin))`，margin 根据偏好强度调整

### 4.4 PPO 的问题

- **训练不稳定**：reward hacking（模型学会讨好奖励模型而非真正变好）
- **超参数敏感**：clip range、KL 惩罚系数需要仔细调
- **计算开销大**：需要同时维护 4 个模型（policy、reference、reward、value）

## 5. DPO（Direct Preference Optimization）

### 5.1 核心思想

DPO 跳过奖励模型，直接用偏好数据优化策略：

```
RLHF:  偏好数据 → 训练奖励模型 → PPO 优化策略
DPO:   偏好数据 → 直接优化策略（隐式奖励）
```

### 5.2 DPO Loss

```python
# DPO 的损失函数
def dpo_loss(policy_chosen, policy_rejected, ref_chosen, ref_rejected, beta=0.1):
    log_ratio_chosen = policy_chosen - ref_chosen
    log_ratio_rejected = policy_rejected - ref_rejected
    loss = -F.logsigmoid(beta * (log_ratio_chosen - log_ratio_rejected))
    return loss.mean()
```

### 5.3 DPO vs RLHF 对比

| 维度 | RLHF (PPO) | DPO |
|------|-----------|-----|
| 实现复杂度 | 高（4 个模型） | 低（2 个模型） |
| 训练稳定性 | 差（reward hacking） | 好 |
| 计算开销 | 大 | 小 |
| 效果 | 强（迭代优化） | 接近 RLHF |
| 超参数 | 多且敏感 | 少（主要是 beta） |

**面试回答要点**：DPO 是当前工业界对齐的首选方案，因为它简单、稳定、效果好。RLHF 更适合需要迭代优化的场景（如 Llama 2 的两阶段 RLHF）。

## 6. 其他对齐方法

### 6.1 RLAIF（AI 反馈强化学习）

用 AI 模型代替人类标注偏好数据：

```
RLHF:  人类标注 A > B
RLAIF: GPT-4 标注 A > B
```

**优势**：标注成本低、一致性高、可扩展
**风险**：AI 偏见可能被放大

### 6.2 Constitutional AI（Anthropic）

1. 让模型自我批评（"这个回答是否有害？"）
2. 用自我批评的结果作为偏好数据
3. 用 RLAIF 训练

### 6.3 ReST（Reinforced Self-Training）

1. 用当前模型生成多个回复
2. 用奖励模型筛选高质量回复
3. 用筛选后的数据做 SFT
4. 重复迭代

## 7. 微调实战 Checklist

### 微调前

- [ ] 明确任务目标（分类/生成/对话）
- [ ] 准备高质量训练数据（质量 > 数量）
- [ ] 选择合适的基础模型（根据任务复杂度和资源）
- [ ] 评估是否真的需要微调（prompt engineering 可能就够了）

### 微调中

- [ ] 从 LoRA r=16, alpha=32 开始
- [ ] 训练所有线性层（不只是 Q/V）
- [ ] 学习率 2e-4，1-2 epochs
- [ ] 监控 loss 曲线，过拟合立即停止
- [ ] 保留验证集评估

### 微调后

- [ ] 在测试集上评估（不要只看 loss）
- [ ] 检查是否出现灾难性遗忘
- [ ] A/B 测试对比基线
- [ ] 记录实验配置（wandb/mlflow）

## 8. 常见踩坑总结

| # | 踩坑 | 正确做法 |
|---|------|---------|
| 1 | rank 设太小（r=4） | 至少 r=16，复杂任务 r=32+ |
| 2 | 只训练 Q/V 层 | 训练所有线性层 |
| 3 | 训练太多 epochs | 1-2 epochs 足够 |
| 4 | alpha = rank | alpha = 2 × rank |
| 5 | 学习率太低 | LoRA 用 2e-4，不是 2e-5 |
| 6 | 数据量太少就微调 | < 100 条数据不如用 few-shot |
| 7 | 忽略数据质量 | 100 条高质量 > 1000 条低质量 |
| 8 | 全量微调小模型 | 小模型用 LoRA，大模型才考虑全量 |
| 9 | 不监控 eval loss | 必须有验证集，否则无法发现过拟合 |
| 10 | DPO 不用 reference model | 必须冻结一份 reference model 计算 KL |

## 9. 面试题

### Q1: LoRA 的原理是什么？为什么有效？

**答**：LoRA 基于"微调时权重变化是低秩的"假设。它冻结原始权重 W，用两个低秩矩阵 B 和 A 来近似权重更新 ΔW = BA，其中 r << d。这样可训练参数从 d² 降到 2dr，通常只有 0.1%-1%。有效的原因是预训练模型已经学到了丰富的表示，微调只需要在一个低维子空间中调整。

### Q2: QLoRA 和 LoRA 的区别？什么时候用 QLoRA？

**答**：QLoRA 在 LoRA 基础上将基础模型量化为 4-bit（NF4 格式），显存节省约 33%，但训练速度慢约 39%。适用于显存受限的场景（如单张 24GB GPU 微调 7B 模型）。如果显存充足，直接用 LoRA + bf16 更快。

### Q3: LoRA 的 rank 怎么选？

**答**：一般从 16 开始。简单任务（情感分类）r=8 足够，复杂任务（代码生成）r=32 或更高。rank 越大表达能力越强，但容易过拟合。经验法则：如果增加 rank 后验证集指标不再提升，说明 rank 已经足够大了。

### Q4: RLHF 和 DPO 怎么选？

**答**：DPO 是当前首选，因为它简单、稳定、只需 2 个模型（policy + reference），不需要训练奖励模型。RLHF 更适合需要迭代优化的场景（如 Llama 2 的两阶段训练），或者当偏好数据质量不高时（奖励模型可以更好地泛化）。

### Q5: 什么是 reward hacking？怎么解决？

**答**：Reward hacking 是指模型学会讨好奖励模型而非真正变好。例如模型发现输出更长的回答能获得更高分数，就无意义地拉长输出。解决方案：(1) KL 散度惩罚，限制策略偏离参考模型的程度；(2) 多个奖励模型取交集；(3) 定期更新奖励模型。

### Q6: 微调和 RAG 怎么选？

**答**：RAG 适合需要外部知识的场景（如企业知识库问答），微调适合需要改变模型行为的场景（如输出格式、风格、领域专业知识）。两者可以结合：先微调让模型适应领域，再用 RAG 注入具体知识。

### Q7: SFT 数据量多少合适？

**答**：取决于任务复杂度。Llama 2 论文表明，高质量的 10K 数据就能获得不错的 SFT 效果。Alpaca 用 52K 数据训练了 7B 模型。一般来说：简单任务 1K-10K，复杂任务 10K-100K。质量远比数量重要——100 条精标注数据 > 1000 条 GPT-4 生成的数据。

### Q8: 如何判断微调是否成功？

**答**：三个维度：(1) 目标指标提升（准确率/BLEU/人工评估）；(2) 未出现灾难性遗忘（在通用基准上性能下降 < 5%）；(3) A/B 测试胜率 > 50%。不能只看 training loss——loss 下降不代表任务指标提升。

### Q9: Prefix Tuning 和 LoRA 的区别？

**答**：Prefix Tuning 在每层注入可训练的前缀向量（类似虚拟 token），不改变模型权重。LoRA 通过低秩矩阵修改权重。LoRA 效果更好且推理时无额外开销（可合并回原权重），是当前工业界首选。Prefix Tuning 的优势是不修改模型结构，适合模型即服务（MaaS）场景。

### Q10: 多任务微调怎么处理？

**答**：常见方案：(1) 每个任务一个 LoRA adapter，推理时动态切换；(2) 多任务混合训练，用数据混合比例控制；(3) LoRA 合并——将多个 LoRA 权重线性组合。推荐方案 1，因为它最灵活且不会互相干扰。
