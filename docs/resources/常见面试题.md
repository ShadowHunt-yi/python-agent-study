# Python Agent 开发常见面试题

## 一、Python 基础

### 1.1 list 和 tuple 的区别？
- list 可变，tuple 不可变
- tuple 可以作为 dict 的 key，list 不行
- tuple 占用内存更小

### 1.2 dict 的底层实现？
- 哈希表（hash table）
- Python 3.7+ 保证插入顺序
- 查找/插入/删除平均 O(1)

### 1.3 GIL 是什么？对 Agent 开发有什么影响？
- Global Interpreter Lock，全局解释器锁
- 同一时刻只有一个线程执行 Python 字节节码
- I/O 密集型任务（如调用 LLM API）不受影响——I/O 操作在 C 层面会释放 GIL，`await` 让事件循环可以调度其他协程
- CPU 密集型任务需要用 `multiprocessing`（多进程，每个进程独立 GIL）
- Agent 开发主要是 I/O 密集（等待 LLM API 响应），所以 `asyncio` 是最佳方案

### 1.4 装饰器的本质是什么？
- 一个接收函数、返回新函数的高阶函数
- `@decorator` 是 `func = decorator(func)` 的语法糖
- 带参数的装饰器是三层函数嵌套

### 1.5 生成器和列表的区别？
- 生成器惰性求值，列表立即求值
- 生成器只能遍历一次，列表可重复遍历
- 生成器省内存，适合大数据集

### 1.6 `async def` 和 `def` 在 FastAPI 中的区别？
- `async def`：直接在事件循环中运行，适合 I/O 操作（调 API、查数据库）
- `def`：FastAPI 会自动放在线程池中运行，适合 CPU 密集操作
- 混用陷阱：在 `async def` 中调用同步阻塞函数（如 `time.sleep`）会阻塞整个事件循环
- 规则：涉及 `await` 用 `async def`，纯同步用 `def`

### 1.7 `yield` 在 Python 中有几种用途？
1. **生成器**：`yield` 暂停函数，下次从暂停处继续（惰性序列）
2. **上下文管理器**：`@contextmanager` + `yield`，`yield` 前是 `__enter__`，后是 `__exit__`
3. **pytest fixture**：`yield` 前是 setup，后是 teardown
4. **FastAPI 依赖**：`yield` 前是获取资源，后是清理资源

### 1.8 `nonlocal` 和 `global` 的区别？
- `global`：声明变量来自模块级作用域
- `nonlocal`：声明变量来自最近的外层函数作用域
- JS 对比：JS 闭包自动捕获外层变量（按引用），Python 的 `nonlocal` 是显式声明
```python
def outer():
    count = 0
    def inner():
        nonlocal count  # 必须声明，否则 inner 的 count 是局部变量
        count += 1
    inner()
    print(count)  # 1
```

### 1.9 Python 的 `is` 和 `==` 有什么区别？什么时候用 `is`？
- `==` 比较值是否相等（调用 `__eq__`）
- `is` 比较是否是同一个对象（内存地址相同）
- **只有 `None`、`True`、`False` 用 `is`**：`if x is None`
- 小整数缓存（-5~256）和字符串驻留可能导致 `is` 对值也返回 True，但这是实现细节，不可依赖

### 1.10 Python 的 `*args` 和 `**kwargs` 是什么？
- `*args`：收集多余的位置参数为 tuple
- `**kwargs`：收集多余的关键字参数为 dict
- 顺序规则：`def f(a, b, *args, kw_only=True, **kwargs)`
- JS 对比：`...args`（rest 参数）对应 `*args`，没有直接对应 `**kwargs` 的语法

### 1.11 Python 的 `match/case` 和 JS 的 `switch` 有什么区别？
- Python 的 `match` 是**模式匹配**，不只是值比较
- 可以匹配结构：解构 tuple、dict、对象属性
- 有守卫条件（`if` guard）
- JS 的 `switch` 只做值比较（`===`），没有解构能力

### 1.12 Python 虚拟环境是什么？为什么需要？
- `venv` 创建独立的 Python 环境，每个项目有自己的 `site-packages`
- 类比 JS：`node_modules` 是项目本地的，Python 默认全局安装
- 没有虚拟环境会：包版本冲突、无法复现环境
- 创建：`python -m venv .venv`，激活：`source .venv/bin/activate`（Linux/Mac）或 `.venv\Scripts\activate`（Windows）

## 二、Pydantic

### 2.1 Pydantic v1 和 v2 的区别？
- v2 用 Rust 核心，性能提升 5-50x
- API 变化：`.dict()` → `.model_dump()`
- `@validator` → `@field_validator`

### 2.2 field_validator 的 mode="before" 和 "after"？
- "before" 在类型转换前执行，拿到原始输入
- "after" 在转换后执行，拿到目标类型

### 2.3 computed_field 的作用？
- 定义派生字段，自动包含在序列化结果中
- 类似 @property 但参与 JSON 序列化

## 三、OpenAI SDK

### 3.1 Tool Calling 的完整流程？
1. 定义工具 schema（JSON Schema）
2. 调用 LLM，传入 tools 参数
3. LLM 返回 tool_calls
4. 解析 tool_calls，执行工具
5. 将工具结果作为 tool message 发回
6. 再次调用 LLM，整合结果

### 3.2 流式输出的实现方式？
- `stream=True`：原始 SSE 事件
- `.stream()`：类型化事件（推荐）

### 3.3 结构化输出怎么实现？
- `response_format` + Pydantic 模型
- `client.beta.chat.completions.parse()` 自动校验

### 3.4 流式输出中如何处理 Tool Calling？
- 流式返回时，`tool_calls` 是分块到达的，需要拼接完整的 JSON
- `choice.delta.tool_calls` 是增量的：每次只返回 `function.arguments` 的一个片段
- 必须等 JSON 拼接完整后才能解析和执行
- 实现：用 buffer 收集 `arguments` 片段，收到 `finish_reason="tool_calls"` 时解析

### 3.5 OpenAI 的 `temperature` 参数怎么调？
- 0：确定性输出，适合代码生成、数据提取
- 0.7：平衡创造性和一致性，适合对话
- 1.0+：高随机性，适合创意写作
- Agent 场景推荐 `temperature=0`，保证工具调用的稳定性

### 3.6 Token 是什么？如何计算成本？
- Token 是 LLM 处理文本的最小单位，大约 1 个英文单词 ≈ 1 token，1 个中文字 ≈ 1.5-2 tokens
- `tiktoken` 库可以计算 token 数
- 成本 = 输入 token × 输入单价 + 输出 token × 输出单价
- Agent 的 token 消耗：系统提示 + 工具定义 + 对话历史 + 工具结果，随轮次增长

## 四、LangChain / LangGraph

### 4.1 LangChain 和 LangGraph 的关系？
- LangChain 提供基础组件（@tool、LCEL、模型集成）
- LangGraph 提供图编排能力
- 生产推荐 LangGraph

### 4.2 LCEL 是什么？
- LangChain Expression Language
- 用 `|` 管道符组合 Runnables
- 提供声明式的流式、异步、批处理能力

### 4.3 LangGraph 的 Reducer 是什么？
- 定义状态更新如何合并
- `Annotated[list, operator.add]` = 追加到列表
- 默认 = 替换

### 4.4 Checkpointing 有什么用？
- 保存图状态快照
- 支持对话持久化、时间旅行调试、中断后恢复

### 4.5 ReAct 循环的工作原理？
1. LLM 思考（Reasoning）
2. LLM 决定调用工具（Action）
3. 执行工具，获取结果（Observation）
4. 将结果反馈给 LLM
5. 重复 1-4 直到 LLM 给出最终答案

### 4.6 Agent 和 Chain 的区别？
- **Chain**：固定流程，预定义步骤序列（A → B → C）
- **Agent**：动态决策，LLM 根据当前状态决定下一步做什么
- 类比：Chain 是流水线，Agent 是有自主判断能力的工人
- 选择：流程固定用 Chain，需要动态决策用 Agent

### 4.7 LangGraph 的 `interrupt` 和 `Command` 是怎么工作的？
- `interrupt(value)`：暂停图执行，将 `value` 返回给调用方，等待人工输入
- `Command(resume=value)`：恢复图执行，将 `value` 作为 `interrupt()` 的返回值
- 流程：图执行到 `interrupt` → 暂停 → 外部获取暂停信息 → 人工处理 → `Command(resume=)` 恢复
- 用途：敏感操作确认、人工审核、分步审批

### 4.8 LangGraph 中 State、Store、Memory 的区别？
- **State**：图的运行时状态（TypedDict），每个节点可以读写，随 Checkpoint 持久化
- **Store**：跨对话的共享存储（KV），适合存放用户偏好、长期记忆
- **Memory**：概念层面，短期=State中的messages，长期=Store中的向量化记忆

### 4.9 多 Agent 协作有哪些模式？
1. **Supervisor**：主管 Agent 分配任务给子 Agent，汇总结果
2. **Swarm/Handoff**：Agent 之间直接交接控制权（A → B → A）
3. **Pipeline**：流水线模式，A 的输出是 B 的输入
4. **Debate**：多个 Agent 辩论，投票决定最终答案
- LangGraph 实现：StateGraph + 条件边 + 共享 State

### 4.10 如何防止 Agent 陷入无限循环？
- 设置 `recursion_limit`（LangGraph）或 `max_iterations`
- 检测重复工具调用（相同工具 + 相同参数）
- 超时机制（整体超时 + 单步超时）
- 设计明确的终止条件（LLM 返回最终答案而非工具调用）

## 五、RAG（检索增强生成）

### 5.1 RAG 的完整流程是什么？
```
文档加载 → 文本分块(Chunking) → 向量化(Embedding) → 存入向量数据库
查询时：用户问题 → 向量化 → 相似度检索 → 拼接上下文 → LLM 生成答案
```

### 5.2 什么是 Embedding？为什么需要？
- Embedding 是将文本映射为高维向量（如 1536 维），向量之间的距离代表语义相似度
- 为什么需要：关键词搜索无法理解语义（"退款" 和 "退钱" 关键词不同但语义相同）
- 模型选择：OpenAI `text-embedding-3-small`、`text-embedding-3-large`，开源 `bge-m3`

### 5.3 文本分块策略有哪些？
- **固定大小**：按字符数切分（如 500 字符 + 50 重叠），简单但可能切断语义
- **递归字符**：按 `\n\n` → `\n` → ` ` 递归切分，保持段落完整性
- **语义分块**：用 Embedding 判断语义边界，质量最好但慢
- **文档结构**：按 Markdown 标题、HTML 标签切分
- `chunk_size` 和 `chunk_overlap` 怎么调？chunk_size 一般 500-1500，overlap 10%-20%

### 5.4 向量数据库选型？
| 数据库 | 特点 | 适用场景 |
|--------|------|---------|
| FAISS | Facebook 开源，纯内存 | 原型开发、小数据集 |
| Chroma | 轻量级，嵌入式 | 本地开发、小项目 |
| Milvus | 分布式，生产级 | 大规模生产环境 |
| Pinecone | 全托管 SaaS | 不想运维 |
| Qdrant | Rust 实现，高性能 | 性能敏感场景 |

### 5.5 RAG 的常见优化手段？
1. **Query 改写**：HyDE（先让 LLM 生成假设答案，用假设答案去检索）、Multi-Query（一个问题生成多个变体）
2. **混合检索**：向量检索 + BM25 关键词检索，取并集
3. **Re-ranking**：用 Cross-Encoder 对检索结果重排序（如 `bge-reranker`）
4. **上下文压缩**：只保留检索结果中与问题相关的部分
5. **Parent-Child Chunk**：小块检索，大块送给 LLM（提高召回率的同时保持上下文）

### 5.6 如何评估 RAG 系统效果？
- **Faithfulness**：答案是否忠实于检索到的上下文（不幻觉）
- **Answer Relevancy**：答案是否回答了问题
- **Context Recall**：检索到的上下文是否覆盖了正确答案
- **Context Precision**：检索到的上下文是否精准（不包含无关内容）
- 工具：RAGAS、DeepEval

### 5.7 Agentic RAG 和普通 RAG 的区别？
- **普通 RAG**：单次检索 → 生成，流程固定
- **Agentic RAG**：Agent 决定是否需要检索、检索什么、检索后是否需要再检索
- 优势：可以多跳推理（先查 A，再根据 A 的结果查 B）、可以判断检索结果是否足够
- 实现：Agent 有 `search` 工具，根据需要自主调用

## 六、FastAPI

### 6.1 FastAPI 的优势？
- 基于 Pydantic 自动校验
- 基于类型提示自动生成文档
- 原生异步支持
- 依赖注入系统

### 6.2 依赖注入怎么用？
- `Depends()` 声明依赖
- 支持依赖链、yield 资源管理
- 测试时用 `app.dependency_overrides` 替换
- 原理：FastAPI 解析函数签名的类型提示，自动构建依赖链

### 6.3 流式响应怎么实现？
- `StreamingResponse` + 生成器
- `media_type="text/event-stream"`（SSE）
- Agent 场景：LLM 流式输出 → SSE 推送给前端

### 6.4 SSE（Server-Sent Events）协议格式？
```
data: 第一条消息\n\n
data: 第二条消息\n\n
event: custom\ndata: 带事件类型的消息\n\n
id: 1\ndata: 带ID的消息\n\n
```
- `data:` 是数据字段，`\n\n` 双换行是消息分隔符
- `event:` 自定义事件类型（前端用 `addEventListener` 监听）
- `id:` 消息 ID（断线重连时用 `Last-Event-ID` 头告知服务器）
- JS 端：`new EventSource("/stream").onmessage = (e) => console.log(e.data)`

### 6.5 中间件（Middleware）和依赖注入（Depends）的区别？
- **中间件**：作用于所有请求/响应的生命周期钩子（CORS、日志、认证）
- **依赖注入**：路由级别的精细控制，只对使用了 `Depends()` 的路由生效
- 选择：全局功能用中间件，路由级功能用依赖注入

### 6.6 Pydantic 验证失败时返回什么？
```json
{
  "detail": [
    {
      "type": "missing",
      "loc": ["body", "name"],
      "msg": "Field required",
      "input": {"age": "not_a_number"}
    }
  ]
}
```
- 状态码 422 Unprocessable Entity
- `loc` 是字段路径，`msg` 是错误描述，`type` 是错误类型
- 前端需要解析 `detail` 数组展示给用户

## 七、asyncio

### 7.1 Task 和协程的区别？
- 协程惰性，不主动执行
- Task 创建后立即调度执行
- JS 对比：`asyncio.create_task(coro)` 类似 `Promise.resolve(coro)`——启动但不等待

### 7.2 gather 和 TaskGroup 的区别？
- `gather`：旧 API，异常时其他任务继续运行，需要手动取消
- `TaskGroup`（Python 3.11+）：结构化并发，一个任务异常自动取消其他任务
- `TaskGroup` 用 `except*` 捕获 `ExceptionGroup`（多个异常的容器）
- 推荐：新项目用 `TaskGroup`

### 7.3 为什么需要 Semaphore？
- 控制并发数，防止服务器过载或连接池耗尽
- 典型场景：同时调用 OpenAI API 限制并发数为 10
```python
sem = asyncio.Semaphore(10)
async def call_with_limit():
    async with sem:
        return await client.chat.completions.create(...)
```

### 7.4 Python 的事件循环和 JS 的有什么区别？
- JS：浏览器/Node.js 自动启动事件循环，无需手动管理
- Python：需要显式启动 `asyncio.run(main())`，或在已运行的循环中用 `await`
- Python 的 `asyncio.run()` 不能在已有循环中调用（会报 `RuntimeError`）
- JS 的 `Promise` 类似 Python 的 `Task`，`async/await` 语法几乎一致

### 7.5 如何处理 asyncio 中的超时？
```python
try:
    result = await asyncio.wait_for(slow_coro(), timeout=5.0)
except asyncio.TimeoutError:
    print("超时了")
```
- `wait_for`：给单个协程设超时
- `asyncio.wait(coro, timeout=)`：给一组任务设超时，返回 (done, pending)
- Agent 场景：LLM 调用必须设超时，防止无限等待

### 7.6 `asyncio.Queue` 的典型用途？
- 生产者-消费者模式：一个协程生产数据放入队列，多个协程从队列消费
- Agent 场景：流式 token 队列（LLM 产出 token → Queue → SSE 推送给前端）
- `None` 作为毒丸（poison pill）：放入 `None` 通知消费者退出

## 八、测试

### 8.1 fixture 的 scope 有哪些？
- function（默认）、class、module、session
- scope 越大，fixture 复用越多，但要注意共享状态的副作用

### 8.2 conftest.py 的作用？
- 放置共享 fixture 和 hooks
- 自动被同级及子目录的测试发现（pytest 的目录发现机制）
- 类比：相当于自动 import 的公共模块

### 8.3 怎么测试 Agent？
- **第一层**：单元测试工具函数（不涉及 LLM）
- **第二层**：Mock LLM 测试决策逻辑（模拟 LLM 返回特定的 tool_calls）
- **第三层**：端到端测试（可选，调用真实 LLM，验证整体流程）

### 8.4 `patch()` 的目标路径怎么写？
- **必须写函数被导入的地方，不是定义的地方**
- 如果 `agent.py` 有 `from openai import AsyncOpenAI`，patch 目标是 `"mini_agent.agent.AsyncOpenAI"`，不是 `"openai.AsyncOpenAI"`
- 常见错误：patch 了定义处的路径，但测试代码导入的是另一处，导致 mock 不生效

### 8.5 `AsyncMock` 和 `MagicMock` 的区别？
- `MagicMock`：模拟同步函数
- `AsyncMock`：模拟 `async def` 函数，支持 `await`
- 用错会报 `TypeError: object MagicMock can't be used in 'await' expression`
```python
from unittest.mock import AsyncMock
mock_llm = AsyncMock()
mock_llm.chat.completions.create.return_value = fake_response
result = await mock_llm.chat.completions.create(...)  # 正确
```

### 8.6 pytest 的 `parametrize` 怎么用？
```python
@pytest.mark.parametrize("input,expected", [
    ("hello", "HELLO"),
    ("world", "WORLD"),
    ("", ""),
])
def test_upper(input, expected):
    assert input.upper() == expected
```
- 一组参数生成多个测试用例
- Agent 测试场景：同一工具函数不同输入的测试

## 九、Prompt Engineering

### 9.1 常见的 Prompt 优化技巧？
- **Few-shot**：给几个输入输出示例，LLM 照着格式回答
- **Chain-of-Thought (CoT)**：要求 LLM "一步一步思考"，提高推理准确率
- **Role Playing**：给 LLM 设定角色（"你是一个资深 Python 工程师"）
- **输出格式约束**：指定 JSON Schema、Markdown 格式
- **System Prompt**：设定全局行为规则（工具使用规范、回答风格）

### 9.2 如何防止 Prompt Injection？
- 输入过滤：检测用户输入中的指令注入（如 "忽略之前的指令"）
- 角色隔离：System Prompt 和 User Input 明确分离
- 输出校验：用 Pydantic 验证 LLM 输出是否符合预期格式
- 最小权限：Agent 的工具只开放必要的能力

### 9.3 System Prompt 设计的最佳实践？
- 明确角色和任务范围
- 列出可用工具及使用条件
- 指定输出格式
- 设定边界（"如果不确定，说不知道"）
- 长度控制：太长消耗 token，太短约束不够

## 十、Agent 架构与工程

### 10.1 Agent 的核心组件是什么？
```
Agent = LLM（大脑） + Tools（手脚） + Memory（记忆） + Planning（规划）
```
- LLM：推理和决策
- Tools：执行具体操作（搜索、计算、API 调用）
- Memory：短期（对话历史）+ 长期（用户偏好、知识库）
- Planning：任务分解、执行计划

### 10.2 如何设计一个好的 Tool？
- **名称清晰**：`search_order` 而非 `do_something`
- **描述准确**：docstring 就是 LLM 看到的工具描述，写清楚"什么时候用"和"参数含义"
- **参数用 Pydantic**：自动生成 JSON Schema，带类型和描述
- **错误处理**：返回错误信息而非抛异常（LLM 可以理解错误并重试）
- **幂等性**：相同输入相同输出，避免副作用

### 10.3 Function Calling 和 MCP 的区别？
- **Function Calling**：OpenAI 提出的标准，每个 LLM 厂商实现略有不同
- **MCP (Model Context Protocol)**：Anthropic 提出的标准化协议，统一工具接口
- MCP 解决的问题：不同 LLM 的 Tool 定义格式不统一，工具需要适配多个 LLM
- 类比：Function Calling 是各家的充电器，MCP 是 USB-C 统一标准

### 10.4 Agent 的可观测性（Observability）怎么做？
- **Trace 追踪**：记录每一步的输入输出（LLM 调用、工具执行、状态变化）
- **工具**：LangSmith（LangChain 官方）、Langfuse（开源）、Phoenix（Arize）
- **关键指标**：延迟、token 消耗、工具调用成功率、循环次数
- **生产建议**：每次 Agent 运行生成 trace_id，关联所有日志

### 10.5 如何控制 Agent 的成本？
- 限制 `max_iterations` 防止无限循环
- 控制对话历史长度（滑动窗口 or 摘要压缩）
- 用小模型做简单任务（路由、分类），大模型做复杂推理
- 语义缓存：相似问题直接返回缓存答案
- 监控每个请求的 token 消耗，设置预算告警

### 10.6 生产环境部署 Agent 的注意事项？
- **超时**：LLM 调用必须设超时（建议 30-60s）
- **重试**：指数退避 + 最大重试次数（应对 429 限流）
- **限流**：控制并发请求数（Semaphore / Redis 限流）
- **降级**：LLM 不可用时的 fallback（缓存答案、转人工）
- **监控**：延迟、错误率、token 消耗、工具调用成功率

## 十一、Transformer 与 LLM 原理

### 11.1 Self-Attention 的计算过程？
1. 每个 token 生成 Q（Query）、K（Key）、V（Value）三个向量
2. 注意力分数 = Q 和 K 的点积，除以 sqrt(d_k) 缩放
3. softmax 归一化为概率分布
4. 输出 = 注意力分数加权的 V
- 公式：Attention(Q,K,V) = softmax(QK^T / sqrt(d_k)) * V
- 复杂度：O(n^2 * d)，n 是序列长度

### 11.2 为什么需要 Multi-Head Attention？
- 单个 attention head 只能关注一种关系
- 多头同时关注不同类型的关系（语法、语义、位置等）
- 计算量和单头相同（总维度不变，每个头维度 = d_model / num_heads）
- 最后拼接所有头的结果，通过线性层投影

### 11.3 位置编码有哪些方案？
- **正弦编码**（原始 Transformer）：sin/cos 函数，不需要训练
- **可学习编码**（BERT）：作为参数训练
- **RoPE**（GPT-NeoX、LLaMA、Claude）：旋转矩阵编码，外推性好
- 位置编码的作用：Transformer 的 Self-Attention 是排列不变的，需要注入顺序信息

### 11.4 Decoder-only 架构为什么成为主流？
- 统一范式：所有任务转化为"预测下一个 token"
- 涌现能力：规模够大时出现 in-context learning、推理等能力
- 训练简单：只需预测下一个 token，不需要复杂的预训练目标
- Scaling Law：参数量和数据量增加时，性能稳定提升

### 11.5 KV Cache 是什么？
- 缓存已计算的 Key 和 Value 矩阵，避免重复计算
- 没有 KV Cache，生成第 n 个 token 需要重新计算前 n-1 个 token 的 attention
- 内存占用：2 × num_layers × num_heads × seq_len × d_head × batch_size
- 是 LLM 推理性能的关键优化

### 11.6 Transformer 和 RNN 的核心区别？
- RNN 顺序处理，长距离依赖丢失，无法并行
- Transformer 用 Self-Attention 并行处理所有 token，直接建模任意距离的依赖
- Transformer 训练更快（GPU 并行），但推理时自回归生成仍是顺序的

## 十二、RAG（检索增强生成）

### 12.1 RAG 的完整流程是什么？
```
文档加载 → 文本分块(Chunking) → 向量化(Embedding) → 存入向量数据库
查询时：用户问题 → 向量化 → 相似度检索 → 拼接上下文 → LLM 生成答案
```

### 12.2 什么是 Embedding？为什么需要？
- Embedding 是将文本映射为高维向量（如 1536 维），向量之间的距离代表语义相似度
- 为什么需要：关键词搜索无法理解语义（"退款" 和 "退钱" 关键词不同但语义相同）
- 模型选择：OpenAI `text-embedding-3-small`、`text-embedding-3-large`，开源 `bge-m3`

### 12.3 文本分块策略有哪些？
- **固定大小**：按字符数切分（如 500 字符 + 50 重叠），简单但可能切断语义
- **递归字符**：按 `\n\n` → `\n` → ` ` 递归切分，保持段落完整性
- **语义分块**：用 Embedding 判断语义边界，质量最好但慢
- **文档结构**：按 Markdown 标题、HTML 标签切分
- `chunk_size` 和 `chunk_overlap` 怎么调？chunk_size 一般 500-1500，overlap 10%-20%

### 12.4 向量数据库选型？
| 数据库 | 特点 | 适用场景 |
|--------|------|---------|
| FAISS | Facebook 开源，纯内存 | 原型开发、小数据集 |
| Chroma | 轻量级，嵌入式 | 本地开发、小项目 |
| Milvus | 分布式，生产级 | 大规模生产环境 |
| Pinecone | 全托管 SaaS | 不想运维 |
| Qdrant | Rust 实现，高性能 | 性能敏感场景 |

### 12.5 RAG 的常见优化手段？
1. **Query 改写**：HyDE（先让 LLM 生成假设答案，用假设答案去检索）、Multi-Query（一个问题生成多个变体）
2. **混合检索**：向量检索 + BM25 关键词检索，取并集
3. **Re-ranking**：用 Cross-Encoder 对检索结果重排序（如 `bge-reranker`）
4. **上下文压缩**：只保留检索结果中与问题相关的部分
5. **Parent-Child Chunk**：小块检索，大块送给 LLM（提高召回率的同时保持上下文）

### 12.6 如何评估 RAG 系统效果？
- **Faithfulness**：答案是否忠实于检索到的上下文（不幻觉）
- **Answer Relevancy**：答案是否回答了问题
- **Context Recall**：检索到的上下文是否覆盖了正确答案
- **Context Precision**：检索到的上下文是否精准（不包含无关内容）
- 工具：RAGAS、DeepEval

### 12.7 Agentic RAG 和普通 RAG 的区别？
- **普通 RAG**：单次检索 → 生成，流程固定
- **Agentic RAG**：Agent 决定是否需要检索、检索什么、检索后是否需要再检索
- 优势：可以多跳推理（先查 A，再根据 A 的结果查 B）、可以判断检索结果是否足够
- 实现：Agent 有 `search` 工具，根据需要自主调用
